# 爬虫服务完整方案与任务说明

## 1. 需求分析与产品原型对应

根据提供的产品原型（`docs/crawler_demo_ui.html`），爬虫服务需重点支持 **运行概览 (Dashboard)**、**网站监控配置 (Tasks)** 和 **采集日志管理 (Logs)** 三大核心模块。

### 1.1 核心模块需求

1.  **运行概览 (Dashboard)**:
    *   **核心指标**: 展示今日 Token 消耗、采集成功率趋势（7天/30天）、异常预警排行。
    *   **数据来源**: 需对日志和任务数据进行聚合统计。

2.  **网站监控配置 (Tasks)**:
    *   **核心场景**: 支持配置**列表页监控**（如新闻列表），自动发现详情页链接并进行增量爬取。
    *   **新建/编辑**:
        *   **基础**: 任务名称、目标 URL、采集模式（单页/全站/列表监控）、监控频率（Cron）。
        *   **规则**: AI 提取 Schema (使用系统默认大模型)。
        *   **高级**: 最大深度、并发数、代理开关、**链接过滤规则**（用于列表页场景）。
    *   **列表管理**: 搜索、状态筛选。
    *   **操作交互**: 立即运行、编辑、查看历史数据、删除、开关定时任务。

3.  **采集日志管理 (Logs)**:
    *   **筛选与导出**: 支持日期范围筛选、状态筛选、CSV 导出。
    *   **详情查看**: 查看系统日志详情及关联的数据项。

## 2. 核心业务流程设计

### 2.1 列表页监控与增量爬取 (核心场景)

针对用户提到的“监控新闻列表页并爬取新增子页面”场景，设计如下逻辑：

1.  **任务触发**: 定时任务触发或用户点击“立即运行”。
2.  **列表页发现 (Discovery)**:
    *   Worker 访问配置的 `url` (列表页)。
    *   **智能/规则筛选**: 提取页面所有链接，根据配置的 `detail_url_pattern` (正则/Glob) 筛选出详情页链接。
    *   **去重 (Deduplication)**: 将筛选出的 URL 与数据库中已存在的 `extract_results.source_url` 比对，过滤掉已爬取的链接。
3.  **增量详情爬取 (Crawl)**:
    *   对**新增**的 URL 列表，生成子任务（或在当前任务中并发执行）。
    *   **内容提取**: 访问详情页 -> 清洗 HTML -> LLM 提取 (基于 Schema) -> 图片转存。
4.  **结果入库**: 存储结构化数据，更新日志。

## 3. 系统架构设计

### 3.1 推荐架构：基于 Redis 的分布式任务队列

为支持高并发和复杂的父子任务（列表->详情），**强烈建议引入 Redis + Celery/ARQ**。

*   **Broker**: Redis (需在 `docker-compose.yml` 中添加)。
*   **Worker**: 独立运行的 Worker 进程。

### 3.2 存储方案

*   **结构化数据**: PostgreSQL (`extract_results` 表存储提取后的 JSONB 数据)。
*   **非结构化数据 (图片)**: MinIO。
    *   **流程**: 下载图片 -> 上传 MinIO -> 替换 Markdown/HTML 中的链接 -> 存入 DB。

### 3.3 数据库模型设计建议

**1. 任务表 (`tasks`) 升级**
需支持更多配置项以匹配 UI。

```python
class Task(Base):
    __tablename__ = "tasks"
    # ... existing fields ...
    
    # 基础配置
    name = Column(String, nullable=False)
    url = Column(String, nullable=False)
    frequency = Column(String)  # Hourly, Daily, Weekly, Manual
    status = Column(String, default="active")  # active, paused
    
    # 爬虫配置
    mode = Column(String, default="scrape")  # scrape(单页), crawl(深度), list(列表监控)
    max_depth = Column(Integer, default=1)
    concurrency = Column(Integer, default=1)
    use_proxy = Column(Boolean, default=False)
    
    # 列表监控专用
    detail_url_pattern = Column(String, nullable=True)  # 正则，用于匹配详情页链接
    
    # AI 提取配置
    schema = Column(JSONB)  # 提取字段定义
    # llm_config 已移除，统一使用系统默认模型
```

**2. 结果表 (`extract_results`)**

```python
class ExtractResult(Base):
    __tablename__ = "extract_results"
    
    id = Column(Integer, primary_key=True)
    task_id = Column(Integer, ForeignKey("tasks.id"))
    job_id = Column(String, index=True)
    
    # 核心元数据
    source_url = Column(String, unique=True)  # 用于去重
    title = Column(String)
    publish_date = Column(DateTime)
    
    # 完整提取数据
    data = Column(JSONB)
    
    created_at = Column(DateTime, default=datetime.utcnow)
```

**3. 日志表 (`logs`) 增强**

## 4. 接口定义 (API Specification)

### 4.1 任务管理 (Tasks)

*   **创建监控任务**
    *   `POST /api/v1/tasks`
    *   **Body**:
        ```json
        {
          "name": "能源局公示监控",
          "url": "https://www.nea.gov.cn/hudong/gongshi/index.htm",
          "frequency": "Hourly",
          "mode": "list",
          "detail_url_pattern": "*/gongshi/202.*.htm",  // 匹配详情页规则
          "schema": { ... },
          // "llm_config": 默认使用系统配置
          "concurrency": 2
        }
        ```
    *   **Response**: `{"id": 1, "status": "created"}`

*   **更新监控配置**
    *   `PUT /api/v1/tasks/{id}`
    *   **Body**: 同创建接口（支持部分更新）。

*   **获取任务列表**
    *   `GET /api/v1/tasks?q=能源&status=active&page=1&size=10`
    *   **Response**: 包含任务详情及 `last_run_time`, `last_items_count`。

*   **立即运行任务**
    *   `POST /api/v1/tasks/{id}/run`
    *   **Desc**: 触发一次手动执行，返回生成的 `job_id`。

*   **切换任务状态 (启动/暂停)**
    *   `POST /api/v1/tasks/{id}/toggle`
    *   **Body**: `{"active": true/false}`
    *   **Desc**: 暂停或恢复定时任务。

*   **删除监控任务**
    *   `DELETE /api/v1/tasks/{id}`
    *   **Desc**: 物理删除任务及关联日志（可选保留日志）。

*   **获取任务历史数据**
    *   `GET /api/v1/tasks/{id}/results?page=1&size=20`
    *   **Response**: 返回 `ExtractResult` 列表。

### 4.2 运行概览 (Dashboard)

*   **获取统计数据**
    *   `GET /api/v1/dashboard/stats`
    *   **Response**: `{"today_tokens": 128420, "total_items": 5430, "error_count": 12}`

*   **获取趋势图表**
    *   `GET /api/v1/dashboard/chart?period=7d`

*   **获取异常排行**
    *   `GET /api/v1/dashboard/alerts`

### 4.3 日志管理 (Logs)

*   **获取日志列表**
    *   `GET /api/v1/logs?start_date=2023-01-01&end_date=2023-01-31&status=fail`

*   **导出日志**
    *   `GET /api/v1/logs/export`

## 5. 开发计划 (Roadmap)

### Phase 1: 核心重构与数据模型 (当前优先级)
1.  **模型更新**: 更新 `Task`, `Log` 表结构，创建 `ExtractResult` 表。
2.  **MinIO 集成**: 实现图片下载、上传与链接替换工具函数。
3.  **列表监控逻辑**: 实现“列表发现 -> 去重 -> 详情爬取”的 Worker 逻辑。

### Phase 2: 接口与业务逻辑
1.  **完善 Task 接口**: 实现 CRUD 及 `run`, `toggle` 接口。
2.  **实现 Dashboard 聚合查询**: 编写 SQL 聚合语句。
3.  **Result 接口**: 实现数据查询与 CSV 导出。

### Phase 3: 稳健性升级
1.  **Redis 队列**: 迁移 `asyncio.Queue` 到 Redis。
2.  **反爬策略**: 实现代理池对接与并发控制优化。
